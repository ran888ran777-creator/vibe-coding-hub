## 🧭 Цель

Собрать маленький, понятный и полезный проект, который удобно развивать вместе с AI-ассистентом. Не «идеальную платформу», а рабочую базу, которую можно улучшать итеративно.

## ⚡ Минимальный стек

| Компонент | Выбор |
| --- | --- |
| Репозиторий | GitHub |
| Сайт | Hugo Extended |
| Хостинг | GitHub Pages |
| Контент | Markdown с YAML front matter |
| AI-правила | `CLAUDE.md` в корне |

## 🔧 Шаги

### 1. Зафиксируйте тему проекта

Определите одну фразу, которая описывает ценность репозитория.

Пример:

> Русскоязычный хаб по vibe coding, Hugo, MCP и AI-воркфлоу.

### 2. Создайте каркас

Минимум, который должен быть с первого дня:

- `hugo.toml`
- `content/`
- `layouts/`
- `static/css/`
- `.github/workflows/`
- `README.md`
- `CLAUDE.md`

### 3. Добавьте правила для ассистента

Хороший AI-friendly репозиторий почти всегда содержит:

- структуру каталогов;
- формат front matter;
- требования к стилю;
- список запретов;
- ожидания по итоговому ответу и проверкам.

### 4. Начните не с десяти статей, а с трёх

Оптимальный стартовый набор:

- один концепт;
- один прикладной гайд;
- одна инструментальная страница.

Так вы быстрее увидите, как работает навигация, стиль и внутренняя связность.

### 5. Настройте автоматический деплой

Для GitHub Pages достаточно workflow, который:

1. ставит Hugo Extended;
2. собирает сайт;
3. генерирует артефакт;
4. деплоит в Pages.

Если нужен поиск, добавьте Pagefind как пост-шаг после `hugo --minify`.

## 💡 Паттерн развития

Рабочий цикл для vibe coding проекта обычно такой:

1. Добавить один полезный материал.
2. Проверить, как он смотрится на главной и в разделе.
3. Улучшить шаблоны только там, где стало тесно.
4. Повторить.

## ⚠️ Частые ошибки

- Слишком рано усложнять архитектуру.
- Подключать много сервисов до первой полезной публикации.
- Пытаться сделать идеальную taxonomy-систему в первый день.
- Не документировать правила для AI-ассистента.
