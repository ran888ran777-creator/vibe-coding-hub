## 🧭 Что это

Claude Code удобен там, где нужен агентный цикл прямо в локальном репозитории: читать код, предлагать изменения, запускать команды и фиксировать рабочие шаги без переключения между браузером и IDE.

## 📦 Установка

Базовый сценарий на Windows:

```powershell
winget install Git.Git
winget install OpenJS.NodeJS.LTS
npm install -g @anthropic-ai/claude-code
```

Проверьте версию:

```powershell
claude --version
```

## ⚙ Базовый конфиг

Минимальный набор того, что стоит определить заранее:

| Параметр | Зачем нужен | Пример |
| --- | --- | --- |
| `ANTHROPIC_API_KEY` | Аутентификация | Переменная окружения в профиле |
| Рабочая папка | Контекст для агента | Корень репозитория |
| Правила проекта | Ограничения и стиль | `CLAUDE.md` в корне |
| Git-стратегия | Контроль изменений | Отдельная ветка под задачу |

Пример установки ключа в PowerShell:

```powershell
setx ANTHROPIC_API_KEY "your_api_key"
```

После перезапуска терминала агент сможет работать без ручного ввода ключа в каждой сессии.

## 🧠 Рабочие паттерны

### 1. Исследование перед правкой

Сначала попросите агента найти точки входа, конфиги и зависимые файлы. Это уменьшает риск точечных правок не в том месте.

Пример запроса:

```text
Просмотри структуру проекта, найди все файлы, связанные с деплоем Hugo, и предложи план без изменений.
```

### 2. Маленькие атомарные изменения

Лучше несколько узких задач, чем одна расплывчатая. Claude Code устойчивее работает, когда зона ответственности ограничена одним сценарием.

Пример:

```text
Обнови только GitHub Actions workflow под GitHub Pages и не трогай контент.
```

### 3. Явные критерии готовности

Если нужно получить законченную задачу, перечислите, что считать готовым:

- какие файлы должны измениться;
- какие проверки надо запустить;
- какие зависимости нельзя добавлять;
- в каком формате вернуть результат.

## 🧱 Ограничения

- Агент не заменяет ручной архитектурный контроль.
- Если правила проекта не описаны, стиль правок начинает плавать.
- Большие запросы с несколькими целями чаще ломают формат ответа.
- Деплой и CI надо проверять отдельно, особенно если пайплайн зависит от секретов и внешних прав.

## 🔐 Практика безопасности

- Давайте доступ только к нужному репозиторию.
- Не просите агента вставлять секреты прямо в код.
- Храните правила доступа и стиль изменений в `CLAUDE.md`.
- Перед коммитом просматривайте дифф, даже если результат выглядит корректным.

## ✅ Когда использовать

- Нужно быстро навести порядок в статическом сайте.
- Требуется собрать boilerplate под Hugo, AI workflow или knowledge hub.
- Нужен помощник, который умеет читать репозиторий, а не только генерировать текст.

## 🚫 Когда не использовать

- Если задача требует долгой продуктовой проработки без чётких критериев.
- Если репозиторий содержит чувствительные данные и нет ясной модели доступа.
- Если нужен только единичный Markdown-ответ без работы с кодовой базой.
